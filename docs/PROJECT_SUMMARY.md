# 项目总结

## 1. 项目定位

本项目是一个用于验证“AI 空间设计前期工作流”的流程原型，同时也包含一个可复用的单代理运行时基础层。它把设计任务书解析、风格生成、视觉提示词编译、效果图任务编排、看板展示和结果落盘串成了一条最小可运行链路。

从工程角度看，这个仓库不是单点功能，而是一个由运行时、专业工作流、展示层和评测层组成的实验平台。

## 2. 核心组成

### `src/agent_runtime`

提供通用 agent runtime 能力，包括：

- agent 注册与隔离运行
- 会话快照与 checkpoint 保存
- 工具调用协议与执行
- 事件记录与可观察性
- 项目看板快照与历史索引

这部分更偏“基础设施层”，可被其他 agent/workflow 复用。

### `design_workflow`

提供空间设计领域工作流，包括：

- orchestrator 阶段编排
- `req_parser`、`material_style`、`visual_prompt` 等 specialist
- prompt 规范、规则和图像生成工具
- full demo 扩展模块，如案例研究、概念提炼、分区、汇报与预算

这部分更偏“业务流程层”。

### `dashboard` + `server.py`

提供本地化的运行看板与 HTTP 接口：

- 查看当前项目状态与历史运行
- 读取效果图结果
- 启动实际工作流
- 支持本地讨论区、轮询与任务状态展示

这部分构成了演示入口和人工观察界面。

### `evaluation` / `tests`

提供两类质量保障：

- 单元测试与集成测试，验证 runtime、workflow、前端资源、PPT 生成等核心能力
- fixture 驱动的离线评测，验证 specialist 输出字段完整性

## 3. 当前可运行能力

最清晰的主链路是 MVP：

1. 输入中文任务书
2. 解析结构化需求
3. 生成材料和风格建议
4. 生成视觉 prompt
5. 编排图片生成任务
6. 把结果落盘到 `output/`
7. 通过看板页面展示当前运行状态

完整 Demo 在结构上已经连通更多 specialist，但整体仍偏原型验证性质。

## 4. 项目亮点

- 运行时与业务流程分层明确，后续扩展成本较低
- 产物、状态、日志和看板之间有关联，便于追踪
- 既有最小可运行 MVP，也保留了 full workflow 的扩展位
- 测试覆盖面不错，包含 runtime、API、看板资源、评测和 demo 级别校验

## 5. 当前边界与风险

- 项目仍以原型和流程验证为主，不是生产级交付系统
- 部分 full workflow specialist 仍是占位/雏形实现
- 真实 LLM 与生图能力依赖外部 API 和密钥配置
- 如果直接公开仓库，必须避免把真实密钥、输出产物和缓存文件一并提交

## 6. 适合的后续方向

- 补齐环境变量加载与配置管理
- 明确 full workflow 的输入/输出 contract
- 增加 benchmark 数据集和质量门槛
- 把项目级产物目录规范进一步落实到全流程
- 在看板中增强运行失败诊断与人工确认节点

## 7. 一句话结论

这是一个把“Agent Runtime + 空间设计工作流 + 本地可视化演示”组合在一起的流程验证项目，适合继续发展成更完整的设计前期自动化平台，也适合作为多阶段 agent 系统的实验骨架。
