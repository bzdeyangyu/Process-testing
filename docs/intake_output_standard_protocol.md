# Intake Output Standard Protocol

## Goal

把五份样本数据正式定义为 V1 的输出标准。以后当用户只提供一段简要设计情况时，系统不能直接给出“完成”结论，而必须先执行标准对比，再决定继续追问、调取资料库，还是输出当前阶段成果。

## Standard Source

- 基准标准：`five-sample-v1`
- 对照对象：`项目任务书`、`场地踏勘资料`、`会议决策回灌`
- 目标不是字数接近，而是结构完整度、证据完整度和可继续推进的生产可用度接近

## Required Runtime Behavior

### 1. 初始接收

用户输入一段简要设计情况后，先生成当前已知信息包：

- 项目类型
- 面积
- 地点
- 预算
- 目标受众
- 风格偏好
- 特殊需求
- 工期或节点

### 2. 标准对比

把当前已知信息对照三段标准输出：

- `项目任务书`
- `场地踏勘资料`
- `会议决策回灌`

系统必须输出：

- `coverage_score`
- `section_scores`
- `missing_critical_items`
- `comparison_summary`

### 3. 缺口处理

如果缺失关键字段，不允许伪造完整成果，必须执行二选一补齐策略：

- 先向用户追问
- 如果企业资料库中已有对应材料，则先调库再补齐

### 4. 继续输出

只有在关键缺口被补齐后，才能输出接近五份样本标准的成果包。若仍未补齐，系统要明确标注：

- 当前成果属于哪一阶段
- 哪些字段来自用户原始输入
- 哪些字段来自资料库
- 哪些字段仍为待补充

## Completion Rule

系统只有在以下条件满足时，才允许宣称“达到标准输出”：

- `项目任务书`不缺关键字段
- `场地踏勘资料`至少具备资料清单与约束信息
- `会议决策回灌`至少具备会议结论或明确标记为尚未发生
- `missing_critical_items`为空
- `coverage_score`达到 85 分及以上

如果不满足，统一进入“继续追问或调库”状态，不得直接宣称完成。

## Current V1 Implementation Mapping

当前 V1 已经在 `req_parser` 阶段落地以下能力：

- 将简要 brief 解析为基础结构化信息
- 自动生成 `output_standard_review`
- 自动生成 `follow_up_questions`
- 自动生成 `recommended_retrievals`
- 自动标记 `source_status`

这意味着系统已经具备“先对照标准，再决定是否继续补资料”的入口能力，但场地资料接入和会议回灌仍需要后续阶段继续实现。
